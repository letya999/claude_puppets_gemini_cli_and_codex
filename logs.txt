  Let me check the Invoke-Flow.ps1 more carefully. The issue with Gemini is "Cannot use both a positional prompt and the --prompt (-p) flag together" - this
   means the script is passing the prompt as positional AND as -p. Looking at the script, line 89:
  $Args = @("-p", $FinalPrompt)
  if ($Model) { $Args += "--model"; $Args += $Model }



  I'm wondering if the model flag placement is causing the prompt to be interpreted as a positional argument, or if Gemini CLI has stricter requirements
  about flag ordering than I expected.

  Actually, I think the real issue might be how PowerShell is passing the arguments when there are newlines or special characters in the prompt text. And
  looking at the role file logic, if the "researcher" role doesn't exist, the system prompt would be empty, but the script would still try to build and pass
   the final prompt with the -p flag.

  $SystemPrompt = if (Test-Path $RoleFile) { Get-Content $RoleFile -Raw } else { "" }

  So if no role file exists, $SystemPrompt is empty, and then $FinalPrompt gets constructed with just the CONTEXT and TASK headers plus the actual task.
  That should work fine with -p $FinalPrompt --model gemini-2.5-pro, but the error about "Cannot use both a positional prompt and the --prompt (-p) flag
  together" suggests something else is being passed as a positional argument somehow—maybe the way @Args is being splatted is causing PowerShell to
  interpret part of the prompt text as a separate positional argument.
